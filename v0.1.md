# Project: Imp

A browser-based AI project manager **and** code-writing agent, powered by the
Claude Code Agent SDK. GitHub Issues + a Projects v2 board are the database.
Claude Code is the brain. The web UI is just a face.

The chat agent's persona is **Foreman** — when the admin opens the web UI they
talk to "Foreman." Under the hood, Foreman drives Imp: it both manages the
project (Gantt, kanban, burndown, delays, dependencies) **and** writes code by
invoking Imp's existing pipeline scripts (moderate issues → solve → fix PRs).

## What This Project Does

Imp does two things on a GitHub repo, through one chat interface:

1. **Project management visibility** — Gantt charts, kanban, burndown,
   dependency analysis, A/B timeline comparisons, "what happens if Issue #42
   slips two weeks" scenarios. The bureaucratic artifacts organizations need
   to believe work is happening.
2. **Code generation** — moderate messy issues into well-formed tasks, write
   code to solve them, open PRs, read review comments, push fixes. This is
   what Imp's existing CLI scripts in `99-tools/` already do; the merged
   project keeps that pipeline and gives it a chat front door.

An admin (the project manager, or whoever owns the GitHub org) clones this repo
and runs `python imp.py`. That starts a long-running web service. The admin
opens it in a browser and is greeted by Foreman in a chat window. A **Setup
Agent** walks them through configuration on first visit (GitHub auth, Claude
auth, repo selection, project board bootstrap, admin password). After that,
every visit goes straight to Foreman.

### Deployment is flexible — VPS *or* laptop

Imp is designed to run in two distinct shapes, and the same codebase supports
both with no conditionals:

- **Always-on (VPS / home server / always-up workstation)** — for projects you
  want Imp watching continuously: the autonomous loop is enabled, Imp moderates
  new issues as they arrive, posts status updates, and is reachable at a stable
  URL behind HTTPS. This is the "invisible PM" mode where contributors see Imp's
  outputs in GitHub but never visit the web UI directly.

- **On-demand (your laptop, while you're doing project work)** — for projects
  you only touch occasionally: clone, `python imp.py`, browser opens, you spend
  30 minutes letting Foreman triage the backlog and open a few PRs, then close
  the laptop. The autonomous loop can stay disabled in this mode; you drive
  every action by hand from chat.

**The pitch for the on-demand mode is the important one**: when you sit down to
review, clean up, or push a project forward, **spinning up Foreman should feel
easier than starting to do the work manually**. The whole point is to lower the
activation energy of "I should look at the GitHub backlog this weekend" until
the cost of opening Foreman is lower than the cost of opening the GitHub
dashboard yourself. If `python imp.py` + a browser tab takes longer than just
doing the triage by hand, Imp has failed at its core job.

This is why setup happens entirely in the browser, why there are no CLI flags
to remember, why the budget controls live in chat, and why there is no
bookkeeping for the admin to do between sessions: Imp tries hard to be the
path of least resistance. Setup runs once per project; after that, every
session is `python imp.py` → tab opens → start chatting.

There are no terminal prompts after `python imp.py`. The terminal only shows
logs. Everything else is the browser.

### CLI mode still exists

The `99-tools/` scripts (`moderate_issues.py`, `solve_issues.py`, `fix_prs.py`,
`run_all.sh`) continue to work standalone for power users, cron jobs, and CI
pipelines. The web UI calls the same scripts as tools — there is no second
implementation. CLI mode and web mode share state (`.imp/`, GitHub labels, the
Projects v2 board), so an admin can run scripts manually from a terminal and
Foreman will see the results next time it syncs, and vice versa. A common
pattern: run Foreman on a VPS for the autonomous loop, and run the same
scripts from CI on demand for one-off batch jobs.

**The contributors on the GitHub project never interact with Foreman directly.**
From their perspective, issues get triaged, dependencies get filled in, comments
ask them to verify estimates, labels appear, milestones get adjusted, PRs open,
status updates get posted — exactly the things a human PM + a junior dev would
do. The fact that an AI is doing it on the admin's behalf is invisible to them.
This is deliberate: Imp should be substitutable for a human PM + assistant
without anyone on the delivery side having to change how they work, or even
know it happened.

## Architecture

Imp uses a **two-layer agent architecture**. A Guard Agent sits between the user
and the worker agent (Foreman) and gates two things:

1. **Inbound user messages** — before the worker ever sees them, to catch prompt
   injection / jailbreak attempts and malicious code in the input.
2. **Outbound write actions** — before any GitHub edit, issue create, label
   change, milestone update, project field update, **or invocation of a code-
   writing pipeline script** (which can result in commits, branches, and PRs)
   is actually executed, to confirm the proposed change is on-task for the
   user's original (approved) request and isn't drift, scope creep, or
   attacker-coerced behavior smuggled in via issue bodies / comments the worker
   read along the way.

The worker agent is the only thing with tools. The guard has none. Nothing reaches
GitHub — neither a label change nor a pull request — without two independent
agents agreeing it should.

There are also two **execution modes** the worker can be in at any moment:

- **Interactive mode** — the admin types in chat, the worker responds. Default.
- **Autonomous loop** — the server triggers the worker on a schedule (or when a
  new issue appears, depending on configuration) to run the moderate → solve →
  fix-PR pipeline without the admin sitting there. The same Guard Agent gates
  every action the autonomous loop produces, and the admin can **pause, resume,
  or scope** the loop from chat at any time (see **Background Processing Loop**
  below).

Chainlit is the entire frontend. We write zero HTML, zero JavaScript, zero
WebSocket protocol. `main.py` consists of Python handlers decorated with
`@cl.password_auth_callback`, `@cl.on_chat_start`, and `@cl.on_message`; those
handlers call Chainlit primitives (`cl.Message`, `cl.Step`, `cl.AskActionMessage`,
etc.) to produce everything the user sees. The security layer (Guard Agent,
checkpoint B interception, budgets) is unchanged from the original design —
it all lives in `server/` and is called from `main.py`'s handlers.

A **proposed action** is the unit checkpoint B operates on. It is one of:

- *Edit issue #N*: title, body, labels, milestone, assignees
- *Update project fields for #N*: e.g. `duration_days=5, source=heuristic`
- *Create issue / Add to project board*
- *Run pipeline*: `moderate_issues.py --issue 42`, `solve_issues.py --issue 42`,
  `fix_prs.py --pr 17` — each invocation is a single proposed action even though
  the underlying script may produce many gh calls, commits, branches, and a PR.
  The script is trusted **once approved**; the trust unit is the invocation.

This level of granularity is deliberate: gating every gh subcommand inside
`solve_issues.py` would mean dozens of LLM round-trips per issue. Gating at the
script-invocation level means one guard call per pipeline run. The guard sees
"the worker wants to run solve_issues.py on issue #42 because the user asked
for that issue to be solved" and approves or rejects on the basis of intent
alignment.

### Layer 1 — Guard Agent (no tools, two responsibilities)

The Guard Agent is a separate Claude session with **no tools** and **no GitHub
access**. It cannot edit anything itself; it can only emit structured verdicts.
The server invokes it at two distinct checkpoints.

**Checkpoint A — inbound user messages**

- Receives the **sanitized** user text (control characters stripped, length capped,
  any embedded markup/HTML rendered inert as plain text, no system-style tags
  allowed through).
- Looks for prompt injection, role-confusion attacks, instruction overrides,
  embedded "ignore previous instructions" payloads, malicious code snippets,
  exfiltration attempts, and clever jailbreak phrasings ("open the claw" / DAN-style
  framings, hypothetical-author tricks, encoded instructions, etc.).
- Returns `{"verdict": "approve" | "reject", "reason": "..."}`.
- On `reject` the worker is **never invoked** for that turn; the UI shows the block
  reason. On `approve` the sanitized text is forwarded to the Foreman Agent and the
  guard records the user's stated intent for use in checkpoint B.

**Checkpoint B — outbound write actions**

- The Foreman Agent does not call `gh issue edit`, `gh issue create`, label changes,
  or milestone updates directly. Those calls are intercepted by the server and
  queued as **proposed actions**.
- Each proposed action is sent to the Guard Agent along with: the original approved
  user request, a short rationale from the worker, and the exact command + payload.
- The guard checks that the proposed edit **actually contributes to fulfilling the
  user's request** — not unrelated cleanup, not drive-by "improvements", not
  changes induced by malicious instructions the worker may have read inside an
  issue body or comment.
- Returns `{"verdict": "approve" | "reject", "reason": "..."}`. Only approved
  actions are executed against GitHub. Rejected actions are reported back to the
  worker so it can revise or abandon them, and surfaced to the user in the UI.

Because the guard has no tools, a compromised or confused worker cannot bypass it,
and a hostile prompt cannot reach the tool-armed worker without being cleared first.

### Layer 2 — Foreman Agent (the worker)

- Only runs on input that passed checkpoint A.
- Has the Claude Code tools — `gh` CLI and `pipeline/*.py` — but every **write**
  through `gh` is routed via the server's action queue and must clear checkpoint B
  before it actually runs. Reads are unrestricted.
- Operates under the edit limiter and the CLAUDE.md instructions described below.

## First Run Experience

`python imp.py` does the **bare minimum** in the terminal: it checks for the
Python dependencies it needs to even start the server (fastapi, uvicorn,
websockets, claude-agent-sdk, argon2-cffi, itsdangerous), and if any are missing
it lists them and asks "Install these? [y/n]" once in the terminal — because
there is no UI yet to ask through. After that it starts the FastAPI server,
prints the URL, and the terminal's job is done. From that point on the terminal
only shows logs.

**Everything else is configured by chatting with a Setup Agent in the browser.**
There is no form-based wizard, no sequence of pages. The admin opens the URL,
gets the same chat UI they will use forever after, and is greeted by a different
agent personality — the **Setup Agent** — whose only job is to talk the admin
through configuration and call tools to actually do it.

### The Setup Agent

The Setup Agent runs whenever `.imp/config.json` is missing or incomplete.
It has its own narrow tool set, distinct from Foreman's tools:

- `set_admin_password(password)` — hash and persist; gates all future access.
- `gh_auth_status()` / `gh_auth_login()` — check and start the device-code flow.
  When `gh_auth_login` runs, the agent surfaces the verification URL + code in
  chat so the admin can complete the GitHub flow in a new tab; the agent polls
  until status flips to authenticated.
- `claude_auth_status()` / `claude_auth_login()` — same pattern for the Claude
  CLI. In `bare` mode (no Claude CLI), the equivalent tool prompts for
  `ANTHROPIC_API_KEY` via the chat's selection/input messages and stores it.
- `list_repos()` — wraps `gh repo list` and returns the repos the authenticated
  account has access to. The agent presents them to the admin (typically via a
  selection message — clickable buttons) and waits for a pick.
- `set_repo(owner_repo)` — validate with `gh repo view` and write to config.
- `list_projects(owner)` — check whether an `Imp` Projects v2 board already
  exists for the chosen owner.
- `create_imp_project()` — runs `gh project create` and then iterates through
  `templates/fields.json` calling `gh project field-create` for each custom
  field defined in the **Issue Schema** section below. Stores the resulting
  project number in `.imp/config.json`.
- `configure_loop(enabled, interval_minutes, max_tasks_per_tick)` — sets the
  autonomous loop's initial config. Default is `enabled=false` so the admin
  has to opt in.
- `mark_setup_complete()` — flips a flag in config that causes the server to
  hand off subsequent sessions to Foreman instead of the Setup Agent.

The Setup Agent uses the **same chat UI, message protocol, and Guard Agent**
as the Foreman Agent — checkpoint A still screens the admin's input, and
checkpoint B still gates write actions (here, "writes" include things like
running `gh auth login`, creating the project board, and writing config). The
admin cannot tell from the UI that they are talking to a different agent; they
just experience a friendly conversational onboarding instead of a form.

If the admin tells the Setup Agent they don't want a project board, the agent
calls `mark_setup_complete()` with a flag indicating **read-only mode**. The
Foreman Agent will then load with project-write tools disabled and a banner in
the chat UI explaining the limitation.

### After setup

Once `mark_setup_complete()` has been called, the server stops loading the
Setup Agent for new sessions and loads the Foreman Agent instead. Visiting the
URL goes straight to the normal chat UI, gated by the admin password. The admin
can re-trigger setup later by clearing the flag (a small "Reconfigure" link in
the UI calls a server endpoint that flips the flag and reloads).

## The Agent's Role (Claude Code System Prompt Context)

When Claude Code runs inside Imp's web service, it adopts the **Foreman** persona
and gets this context via CLAUDE.md:

You are Foreman, an AI project manager and engineering assistant. You manage a
GitHub project at {repo}. You both **report on** the project (charts, status,
delays) and **act on** it (triage issues, write code, open PRs, push fixes).
You have two kinds of tools: read/visibility tools you can use freely, and
write tools whose every invocation is reviewed by a separate Guard Agent before
it actually executes.

### What you can do — read & visibility (no guard checkpoint):

- **Sync**: Run `python pipeline/sync_issues.py` to pull current issue state and
  Projects v2 field values from GitHub. Produces `.imp/issues.json`.
- **Analyze**: Run `python pipeline/heuristics.py` to infer missing data
  (durations, dependencies, blockers, delays). Outputs `.imp/enriched.json`.
  Fields inferred by heuristics are marked `"source": "heuristic"`. Fields you
  infer from reading issue descriptions/comments are marked `"source": "llm"`.
  Fields from GitHub structured data are marked `"source": "github"`.
- **Render**: Run `python pipeline/render_chart.py --template gantt` (or kanban,
  burndown, comparison) to produce an HTML chart from enriched data. Output goes
  to `.imp/output/`. Return the HTML to the user for inline display.
- **Scenario**: Run `python pipeline/scenario.py --delay "Issue #12: +14d"` to
  generate an A/B comparison showing original vs delayed timeline.
- **Read GitHub**: Use `gh issue view`, `gh issue list`, `gh pr view`,
  `gh project item-list` freely. No checkpoint. Reads are unrestricted; the
  security boundary is at writes (the guard's checkpoint B will catch anything
  malicious you read that tries to influence a later write).

### What you can do — project management writes (gated by checkpoint B):

- **Edit issues**: `gh issue edit` to update labels, milestones, assignees,
  titles. Counts toward edit budget.
- **Update project fields**: `gh project item-edit` to set the custom fields
  defined in the **Issue Schema** section below (duration, dates, confidence,
  source, verified, depends_on text). All inferred PM data lives in project
  fields, **not** in issue bodies. Counts toward edit budget.
- **Create issues**: `gh issue create` when the user asks, then add the new
  issue to the Imp project board with `gh project item-add`. Counts toward edit
  budget.

### What you can do — code-writing pipeline (gated by checkpoint B + budgets):

The same scripts the standalone CLI mode uses. Each invocation is one **proposed
action** from the guard's perspective.

- **Moderate an issue**: Run `python 99-tools/moderate_issues.py --issue N` to
  format a messy human-written issue into a well-structured task and apply the
  `llm-ready` label when complete (or ask the human for clarification via a
  comment if the issue is too vague). Counts as one task against the task
  budget. May spend tokens (counted against the token budget).
- **Solve an issue**: Run `python 99-tools/solve_issues.py --issue N` on an
  `llm-ready` issue. The script writes code on a new branch, commits, pushes,
  and opens a PR. Sets `in-progress` while running. One task against the task
  budget; commits/PR creation are part of the trusted unit, not gated
  individually.
- **Fix a PR**: Run `python 99-tools/fix_prs.py --pr N` to read review comments
  on a PR and push fixes to its branch. One task.
- **Run the full loop on a target**: Run `python 99-tools/run_all.sh
  --max-tasks N` to do moderate → solve → fix-PR in sequence. The guard
  approves the *whole loop* as one proposed action; the loop itself respects
  the task budget. Use this when the user says something like "process the next
  3 issues."

When running any of the code-writing scripts, you **must not** wrap them with
extra arguments the user did not authorize. The guard compares the exact command
you propose against the user's stated intent. If the user said "moderate issue
42," proposing `solve_issues.py --issue 42` is a checkpoint-B rejection.

### What you can do — chat / control tools (gated, but lightweight):

These tools control the service itself. The guard approves them based on whether
the user asked for the change.

- **Pause / resume autonomous loop**: `loop_pause()`, `loop_resume()`.
- **Scope autonomous loop**: `loop_scope(only_issues=[42, 43])` /
  `loop_scope(clear=True)`. When scoped, the loop only processes the listed
  issues/PRs and ignores everything else. Useful when the user is in a token
  crunch ("only work on issue 42 for now").
- **Set budgets**: `set_token_budget(N)`, `set_edit_budget(N)`,
  `set_task_budget(N)`. Updates `.imp/config.json` and resets the per-session
  counters if the user says so.
- **Show budgets**: `get_budgets()` returns current usage and remaining for
  all three counters.
- **Reset counters**: `reset_budgets(which=["tokens", "edits", "tasks"])`.

### How you respond:

You emit plain text (markdown). The Python handler in `main.py` that runs you
wraps your output into Chainlit messages and steps — you don't construct those
yourself.

- When the user asks a question about project status, sync first if data is
  stale (older than 1 hour), then answer from enriched data.
- When you produce a chart, emit a **mermaid fenced code block** in your
  response, e.g. ```mermaid\ngantt\n...```. Chainlit renders it inline.
  Kanban is expressed as a mermaid flowchart; burndown as a mermaid pie or
  (for real trend lines) a quickchart.io image URL inside a `cl.Image` element
  the handler adds.
- When you want the user to choose between options, tell the handler by
  emitting a structured marker: the handler watches your output for a
  `<selection options="a|b|c">prompt?</selection>` tag and converts it into a
  `cl.AskActionMessage`. This is the one place you emit structured output the
  handler parses; everything else is plain markdown.
- When showing diffs, use a standard unified-diff fenced code block:
  ```diff\n- old\n+ new\n```. Chainlit renders it with syntax highlighting.
- Tool calls you make (`python pipeline/sync_issues.py`, `gh issue edit ...`,
  `python 99-tools/solve_issues.py --issue 42`) are automatically wrapped in
  `cl.Step` blocks by `intercept.py` so the user sees exactly what's running
  and why. You don't need to explain "now I'm running sync_issues.py" — the
  step shows it.
- For everything else, use markdown.

### Issue Schema

Imp stores all of its inferred PM data in **GitHub Projects v2 custom fields**,
not in issue bodies. Issue bodies stay human-owned. The Imp project board is
provisioned at first run with the following fields (see `templates/fields.json`):

| Field            | Type          | Values / format                       |
|------------------|---------------|---------------------------------------|
| `duration_days`  | Number        | integer days                          |
| `start_date`     | Date          | ISO `YYYY-MM-DD`                      |
| `end_date`       | Date          | ISO `YYYY-MM-DD`                      |
| `confidence`     | Single-select | `high` / `medium` / `low`             |
| `source`         | Single-select | `github` / `heuristic` / `llm`        |
| `assignee_verified` | Single-select | `yes` / `no`                       |
| `depends_on`     | Text          | comma-separated issue refs, e.g. `#12, #15` |

`depends_on` is intentionally a plain text field rather than something structured —
it stays readable in the GitHub UI for humans browsing the board, and the parser
is just "split on commas, strip `#`, ignore anything that isn't an integer". If
the agent or pipeline can't parse a `depends_on` value cleanly, it **logs the
issue and continues** rather than failing the whole sync — that issue is treated
as having no known dependencies for that run, and the user is notified in the UI.

When you infer field values, set `confidence` and `source` appropriately and
create a comment on the issue tagging the assignee asking them to verify. After
human verification, set `assignee_verified` to `yes` and add the
`imp:verified` label.

You read these fields via `gh project item-list --format json` (or the GraphQL
API for richer queries) and write them via `gh project item-edit`. You do **not**
write structured data into issue bodies.

### GitHub Labels

Imp uses two label families. Both must exist in the repo (the worker creates
any that are missing during the first sync after setup).

**Code-writing pipeline labels (from the original 99-tools — DO NOT rename, the
CLI scripts depend on these exact names for backward compatibility):**

- `llm-ready` — Issue is moderated and ready for `solve_issues.py`
- `needs-human` — Issue needs human input; the moderator gave up
- `in-progress` — `solve_issues.py` is currently working on this issue

**Visibility / PM labels (Imp-specific, namespaced):**

- `imp:baseline` — Part of original timeline
- `imp:delayed` — Has been delayed from baseline
- `imp:blocked` — Waiting on external input
- `imp:verified` — Human verified the AI estimates
- `imp:needs-review` — AI filled data, needs human check

### Milestones

## Authentication

Imp's web service is a **single-admin** app deployed persistently. There is
exactly one user: the admin who set it up. Auth is intentionally minimal and
implemented with Chainlit's built-in password callback:

```python
@cl.password_auth_callback
def auth(username: str, password: str) -> cl.User | None:
    cfg = load_config()
    if not cfg.get("admin_password_hash"):
        return None
    if verify_password(password, cfg["admin_password_hash"]):
        return cl.User(identifier="admin", metadata={"role": "admin"})
    return None
```

- The Setup Agent calls `set_admin_password()` on the very first conversation,
  before any other tool. The password is hashed (argon2 via `argon2-cffi`) and
  stored in `.imp/config.json`.
- Chainlit handles the login form, cookie issuing, and session validation —
  we only provide the callback above. The cookie is HTTP-only and signed with
  `CHAINLIT_AUTH_SECRET`, which `imp.py` generates once and persists to
  `.imp/chainlit_secret` (mode 0600).
- `@cl.on_chat_start` and `@cl.on_message` receive the authenticated `cl.User`
  from Chainlit; there's nothing for us to gate manually. If the callback
  returns `None`, Chainlit shows "Invalid credentials" and never calls our
  handlers.
- No signup flow, no password reset, no user table, no roles. If the admin
  forgets the password, the recovery procedure is "SSH into the host and delete
  the `admin_password_hash` field from `.imp/config.json`" — documented in
  `docs/README.md`. On the next start, the Setup Agent will re-run and prompt
  for a new password.
- Because Imp's web service is meant to be exposed beyond localhost, the admin
  is expected to put it behind HTTPS (Caddy / Cloudflare Tunnel / nginx).
  `docs/README.md` shows the recommended Caddyfile. Chainlit does not
  terminate TLS itself.

The CLI mode (`./99-tools/run_all.sh`) has no auth — it relies on filesystem
permissions and the user's own `gh auth` token, exactly as it does today.

## Budgets

Imp tracks **three independent budgets**, all stored in `.imp/state.json`
(`server/budgets.py`, which the legacy `99-tools/_state.py` shims into):

| Budget | Counts | Default | Purpose |
|---|---|---|---|
| **Token budget** | Claude API tokens (in + out), across all agents and pipeline runs | 200,000 | Cost control |
| **Edit budget** | Approved checkpoint-B writes to GitHub | 50 | Mutation rate-limiting / safety |
| **Task budget** | Pipeline-script invocations (`moderate_issues.py`, `solve_issues.py`, `fix_prs.py`) | 10 | Coarse "how much work" knob |

The budgets are independent because they answer different questions: tokens are
"how much money," edits are "how much GitHub mutation am I comfortable with
without checking back in," tasks are "how many big units of work to do this
session." The same `solve_issues.py` invocation typically spends ~1 task,
~10–30 edits, and several thousand tokens — all three need to stay above zero
for it to start.

There are no `--limit` CLI flags on `imp.py`. Budgets are set and reset entirely
through chat with Foreman:

> "Foreman, set the token budget to 100k and reset the edit counter."

The budget tools (`set_token_budget`, `set_edit_budget`, `set_task_budget`,
`reset_budgets`, `get_budgets`) are listed under **chat / control tools** in the
agent role section. The same `.imp/state.json` is read by the standalone CLI
mode, so a budget set via chat applies to subsequent `./99-tools/run_all.sh`
runs and vice versa.

When any budget hits zero, the worker can still read and render charts but
checkpoint B will reject every **new** proposed action until the admin resets
the relevant counter or raises the cap. **A subprocess that's already running
is never killed by a budget tick** — it runs to completion within its own
per-invocation cap, and the budget exhausts "into" it rather than "through" it.
See **Runtime control and observability → Budget exhaustion while a subprocess
is running** for the precise semantics.

Budgets are also enforced **per-invocation**: when the worker proposes
running `solve_issues.py` or one of the other code-writing scripts, it passes
`--max-tokens N` where N is `min(remaining_budget, default_per_invocation_cap)`
so that a single pipeline run can't swallow the entire remaining budget. The
existing 99-tools scripts already honor this flag; see the linked section
above for how `intercept.py` cancels an in-flight run if needed.

The three counters live in the Chainlit sidebar (`cl.ChatSettings`) alongside
the loop controls — see **Background Processing Loop → Pause UX** below.

## Background Processing Loop

Imp can run autonomously: at a configured interval, the server triggers Foreman
to do a moderate → solve → fix-PR pass over the repo without anyone watching.
This is the same pipeline `99-tools/run_all.sh` runs from cron, but driven by
the long-running web service so the admin can pause, scope, and observe it from
chat in real time.

### Loop configuration

The Setup Agent asks during onboarding: *"Do you want Imp to autonomously
process new issues, or only when you ask?"* The answer is stored in
`.imp/config.json` under `loop`:

```json
"loop": {
  "enabled": true,
  "interval_minutes": 60,
  "scope": null,
  "paused": false,
  "max_tasks_per_tick": 3
}
```

- `enabled` — whether the loop runs at all. Default false; the admin opts in.
- `interval_minutes` — how often to wake up. Below 5 minutes is rejected.
- `scope` — `null` means "process anything ready"; otherwise an object like
  `{"only_issues": [42, 43]}` or `{"only_prs": [17]}` restricts the loop.
- `paused` — soft pause; the schedule keeps ticking but each tick is a no-op
  until cleared.
- `max_tasks_per_tick` — caps how many pipeline invocations one tick can
  produce, on top of the global task budget.

### Chat controls

The admin can manage the loop by talking to Foreman. Foreman calls the chat
control tools listed in the agent-role section:

> "Foreman, pause the loop." → `loop_pause()`
> "Resume." → `loop_resume()`
> "Only work on issues 42 and 43 for now." → `loop_scope(only_issues=[42, 43])`
> "Forget the scope." → `loop_scope(clear=True)`
> "Process the next 3 issues right now." → triggers an immediate tick with
> `max_tasks_per_tick=3` regardless of schedule.
> "Stop the loop entirely." → `loop_disable()`, sets `enabled=false`.

Every loop tick runs through the **same Guard Agent**. When the loop wakes up
on its own, the synthetic "user request" passed to checkpoint A is the loop's
configured intent ("autonomously process ready issues, scoped to {scope}");
checkpoint B then evaluates each proposed action against that intent. The
guard never approves a loop-originated action that doesn't fall within the
configured scope, so a malicious issue body cannot expand the loop's reach.

### Pause UX

Loop state lives in Chainlit's **sidebar** via `cl.ChatSettings`. The sidebar
shows four controls:

- A text readout: `▶ running (next tick in 47m)` / `⏸ paused` /
  `🎯 scoped: #42, #43` / `⏹ off`
- A `cl.Switch(id="loop_paused")` — toggle pauses and resumes
- A `cl.TextInput(id="loop_scope")` — accepts `42, 43` or empty to clear
- A `cl.NumberInput(id="loop_interval_minutes")` — minimum 5

Changes in the sidebar are received by `@cl.on_settings_update` and written
to `.imp/config.json`; `server/loop.py` hot-reloads the config on the next
tick. The admin can also pause via chat (`"pause the loop"`) and Foreman
calls the same underlying tool — both paths land on the same config field.

Chainlit does not have a primitive for a status banner *above the chat
input* specifically, so this is the chosen workaround. It's slightly less
immediate than a banner but clicking a sidebar switch is one click and
always visible while scrolled through chat history.

## Action Interception (how checkpoint B is enforced)

Checkpoint B's promise — "no GitHub write executes until the guard approves it"
— is only meaningful if there's a chokepoint the worker physically cannot bypass.
Imp uses two mechanisms in order of preference:

1. **Claude Agent SDK pre-tool-call hook (preferred).** The SDK exposes a
   callback the server registers to intercept every Bash tool call before it
   executes. `server/intercept.py` parses the command. If it's a read, pass
   through. If it's a write (gh subcommand on a known mutating verb, or any
   `99-tools/*.py` invocation), build a *proposed action*, hand it to the guard,
   and either execute the original command or return an error string the worker
   sees as the tool result. The worker has no way to reach Bash that bypasses
   this hook.

2. **`gh` shim on PATH (fallback if the SDK hook is unavailable).** The server
   prepends a per-session directory to the worker subprocess's `PATH` containing
   a `gh` script that classifies the subcommand and forwards via Unix socket to
   `server/intercept.py` for the same approve/execute flow. The real `gh` is
   only invoked from the server side, never directly by the worker. The
   `99-tools/*.py` scripts are wrapped the same way (a shim of the same name in
   the per-session dir).

`server/intercept.py` is the single source of truth for: classifying commands as
read/write, building proposed actions, calling the guard, executing approved
actions, incrementing the edit and task budgets, tracking running subprocesses,
and returning results to the worker. The worker tools listed in the agent role
section are documentation — the enforcement lives in `intercept.py`.

## Runtime control and observability

Once the worker starts running `solve_issues.py` or `fix_prs.py`, the admin
needs three things: (a) a way to see what's happening, (b) a way to stop it,
and (c) guarantees that the various limits mean what they say. This section
spells out the concrete semantics so `server/intercept.py` and the Chainlit
sidebar don't have to invent them at implementation time.

### The three layers of limits

| Layer | What it limits | Where enforced | What happens at exhaustion |
|---|---|---|---|
| **Soft budgets** (tokens / edits / tasks) | Total across the whole session | Checkpoint B in `server/intercept.py` reads `.imp/state.json` | The **next** proposed action is rejected with "budget exhausted". **In-flight tasks are not killed** — they're already approved and finish their own way. |
| **Per-invocation caps** | A single pipeline script's own spending | `--max-tokens N` and `--timeout S` flags passed to `moderate_issues.py` / `solve_issues.py` / `fix_prs.py` | The script aborts itself cleanly via its SIGTERM handler (see below). The existing 99-tools scripts already accept `--max-tokens`. |
| **Hard kill** | One running subprocess right now | `server/intercept.py` sends SIGTERM → 15s grace → SIGKILL | Partial git state may remain. Chat reports what was interrupted and how to clean up. |

The three are a chain: **budgets** decide whether to start an action,
**per-invocation caps** decide how big one action can get, **hard kill** is the
emergency stop.

### Where per-invocation caps come from

When the worker proposes `python 99-tools/solve_issues.py --issue 42`, the
Foreman agent computes the `--max-tokens` value at invocation time as:

```
min(current_remaining_token_budget, default_per_invocation_cap)
```

Where `default_per_invocation_cap` is a constant in `server/budgets.py`
(initial default: 25,000 tokens — about enough for `solve_issues.py` to handle
one medium-complexity issue). The admin can override per-invocation caps in
chat (*"use at most 10k tokens for this one"*) before approving the action.
The guard sees the final command with the flag baked in.

The pipeline script is responsible for honoring the cap — the existing
`_state.py` + `check_budget()` machinery in `99-tools/` already does this for
each script's own Claude runs.

### The three flavors of pause

| Flavor | What it pauses | Scope | Effect on in-flight work |
|---|---|---|---|
| **Loop pause** | The autonomous scheduler (`server/loop.py`) | Background loop only | None — interactive chat still works, running pipeline subprocesses run to completion. Next loop tick is a no-op. |
| **Manual pause** | New actions from the interactive worker | Both ad-hoc and autonomous | The running subprocess finishes. No new `cl.AskActionMessage` is shown until resumed. Chat shows "Pause requested — current task will finish first." |
| **Abort running** | One specific subprocess | Surgical, per-PID | SIGTERM → 15s → SIGKILL. Partial state may remain. |

The first two are flags in `.imp/config.json`:

```json
"loop": { "paused": true, ... },
"accepting_new_actions": false
```

The third is a one-shot action: there's a button/command, it finds the PID in
`intercept.py`'s running-tasks map, and sends SIGTERM.

### The SIGTERM contract for pipeline scripts

All three 99-tools scripts (`moderate_issues.py`, `solve_issues.py`,
`fix_prs.py`) install a SIGTERM handler that does best-effort rollback:

```python
def on_sigterm(signum, frame):
    # Stop any in-flight Claude session
    if claude_session is not None:
        claude_session.close()
    # Roll back git state if we created a branch but haven't pushed/opened a PR
    if branch_created and not pushed:
        subprocess.run(["git", "checkout", original_branch], check=False)
        subprocess.run(["git", "branch", "-D", branch], check=False)
    # Record what got committed to .imp/state.json for budget accounting
    record_partial_run()
    # Tell the parent process (intercept.py) what to show the user
    print(f"INTERRUPTED: branch={branch or 'none'} pushed={pushed} pr={pr or 'none'}",
          file=sys.stderr)
    sys.exit(130)

signal.signal(signal.SIGTERM, on_sigterm)
```

The contract has four rules:
1. **Stop the Claude API session first** so no more tokens are charged.
2. **Roll back local git state** only if nothing has been pushed yet. Once a
   branch is on origin, leave it — the admin can delete it themselves if they
   want. (Pushed-but-PR-not-opened is the weirdest case; prefer to open the PR
   anyway so there's a trail.)
3. **Record partial token/edit counts** to `.imp/state.json` so the budget
   accounting stays honest.
4. **Print a single-line INTERRUPTED summary to stderr** so `intercept.py` can
   parse it and tell the user exactly what's left on disk.

If the script doesn't exit within 15 seconds of SIGTERM, `intercept.py` sends
SIGKILL. SIGKILL can't be caught, so the rollback contract doesn't apply and
the state may be arbitrary. `intercept.py` logs the SIGKILL and tells the user
their clean-up options.

### Budget exhaustion while a subprocess is running

This is a specific edge case that matters enough to spell out:

- At time T, the token budget has 5,000 tokens remaining.
- Admin says "solve issue 42". Worker proposes `solve_issues.py --issue 42
  --max-tokens 5000`. Guard approves. Subprocess starts.
- At T+90s, subprocess has used 4,200 tokens. Still running.
- Admin says "solve issue 43". Worker proposes `solve_issues.py --issue 43
  --max-tokens 800` (the remaining budget *as of the new action's approval*).
- Guard rejects at checkpoint B because 800 is below the minimum viable
  per-invocation cap (a floor defined in `server/budgets.py`, default 2,000
  tokens). Chat: "Not enough token budget to start this task. Current run
  will finish (~800 tokens left), then no more tasks until you raise the
  budget."
- At T+120s, subprocess #42 finishes with 4,800 tokens used. Budget is now
  200 remaining.
- Any new proposed action is rejected at checkpoint B.

Budgets exhaust **into** running subprocesses, not **through** them. A task
already in flight is never killed by a budget tick.

### Observability: what the admin sees while things run

Chainlit owns the presentation; these are the affordances. All are built from
native Chainlit primitives — no custom HTML.

1. **Live subprocess logs inside `cl.Step`.** Every pipeline invocation is
   wrapped in a `cl.Step(name=cmd, type="tool")` async context manager. The
   step's `input` is the command; `output` is updated incrementally via
   `step.update()` as lines arrive on stdout/stderr, so the admin sees
   `solve_issues.py` stream its progress line by line exactly like
   `tail -f`. Collapsed by default, expandable. Tool calls nested inside a
   pipeline (each `gh` call, etc.) become nested `cl.Step` blocks.

2. **Raw log files in `.imp/output/<action_id>.log`.** `intercept.py`
   duplicates every subprocess's combined stdout+stderr into a per-action log
   file. Power users who want to `tail -f` from a separate terminal can do so
   without opening the browser. The files are kept for the session and
   rotated out when `.imp/output/` exceeds a configurable size cap (default
   100 MB). Also useful for post-mortem when something went wrong.

3. **`cl.TaskList` pinned above the chat input.** Shows currently running
   pipeline subprocesses and the ~10 most recently completed ones, with
   status icons (🟢 running, ✓ done, ✗ failed, 🛑 aborted). Always visible
   while you scroll chat history. One click on an entry scrolls the chat to
   the corresponding `cl.Step` block.

4. **`cl.ChatSettings` sidebar.** Already planned for loop controls and
   budget counters (see **Background Processing Loop → Pause UX**). Add:
   - A "Currently running" text field showing the name of the in-flight task
     (or `None`).
   - An "Abort running task" button that only appears when a subprocess is
     tracked in `intercept.py`'s running-tasks map. Clicking it sends SIGTERM.

5. **System event messages.** Guard blocks, budget exhaustions, loop tick
   starts, errors, and aborts all become `cl.Message(author="System", ...)`
   entries in chat with a distinct author so they interleave in chronological
   order with everything else. No separate "events" view.

6. **Chat commands for history and status.** Foreman has tools:
   - *"show recent actions"* → markdown table of the last 20 proposed
     actions with outcomes (from the action queue in `.imp/state.json`)
   - *"show logs for action <id>"* → posts the contents of
     `.imp/output/<id>.log` as a collapsible `cl.Step`
   - *"status"* → one-shot dashboard message with budgets, loop state,
     running tasks, and recent event counts

Put together, this is a dashboard built entirely from native Chainlit
primitives plus two files on disk. No custom HTML, no second UI surface, no
separate log viewer, no JSX.

### The `running_tasks` map in `intercept.py`

The load-bearing data structure for all of the above:

```python
@dataclass
class RunningTask:
    action_id: str
    proc: asyncio.subprocess.Process
    pid: int
    command: list[str]
    started_at: datetime
    step: cl.Step              # live Chainlit step for log streaming
    task_entry: cl.Task         # entry in the cl.TaskList
    log_file: Path              # .imp/output/<action_id>.log

running_tasks: dict[str, RunningTask] = {}
```

Kept in memory in the Chainlit process. Updated when:
- A new approved action starts a subprocess → `running_tasks[id] = ...`
- The subprocess exits naturally → entry removed, step finalized, task_entry marked done
- A SIGTERM comes in (abort button / chat command) → `proc.terminate()`,
  then watchdog sends `proc.kill()` after 15s if still alive
- `intercept.py`'s caller wants a status snapshot → iterate the dict

A single-process design is fine because there's one Chainlit process per Imp
deployment. No IPC, no shared-memory, no database.

## Chainlit UI primitives

There is no custom WebSocket protocol — Chainlit owns the wire. We write
Python handlers against Chainlit's API; everything the user sees is a
primitive Chainlit already provides. This table maps each UI need to the
exact Chainlit call.

| UI need | Chainlit primitive | Example |
|---|---|---|
| Admin sends a chat message | `@cl.on_message` handler | `async def on_msg(msg: cl.Message): ...` |
| Foreman replies with text / markdown | `await cl.Message(content=..., author="Foreman").send()` | streamed via `stream_token` |
| A tool call is visible in chat (sync_issues, solve_issues, gh ...) | `async with cl.Step(name=..., type="tool") as step:` | Sets `step.input` and `step.output`, collapsible in UI |
| Foreman asks the admin to pick between options | `cl.AskActionMessage(actions=[cl.Action(...)])` | Button row below the message |
| Foreman asks for a free-text value (e.g. repo name in Setup) | `cl.AskUserMessage(content=...)` | Returns `{"output": "..."}` |
| Proposed action with approval gate ("run solve_issues.py?") | `cl.Step` (guard verdict) + `cl.AskActionMessage` (human approve/reject) | Two-beat UX: guard decides, then human confirms |
| Chart rendering (Gantt / kanban / burndown) | Mermaid fenced code block in a `cl.Message` | `\`\`\`mermaid\ngantt\n...\`\`\`` — Chainlit renders it |
| Diff rendering | Unified diff in a `diff` fenced code block | Native syntax highlighting |
| File download (e.g. exported Gantt HTML) | `cl.File(path=...)` element | Attached to a message |
| Single-admin password auth | `@cl.password_auth_callback` | Returns `cl.User` on success |
| Checkpoint A block surfaced to admin | `cl.Message(author="Guard", content="Blocked: ...")` | Worker is never invoked that turn |
| Loop status + pause/resume controls | `cl.ChatSettings([cl.Switch(id="loop_paused"...), ...])` | Sidebar, persistent across messages |
| Budget counters | Same sidebar via `cl.ChatSettings` + a status `cl.Message` on changes | Or a pinned message |
| Setup vs. Foreman dispatch | `@cl.on_chat_start` reads `.imp/config.json["setup_complete"]` | Dispatches to setup or foreman handler |
| Persistence of admin's session | `cl.user_session.get/set("key", value)` | In-memory per session |

The Python-side security spine (Guard Agent, checkpoint B interception, budget
tracking, autonomous loop) is unchanged: `server/intercept.py` is called from
inside the `@cl.on_message` handler, the guard returns a verdict, approved
actions run, and the admin sees the outcome as Step updates and subsequent
Messages. Chainlit doesn't know or care that there's a security layer — it
just delivers whatever Python messages/steps we emit.

## Tech Stack

- **Python 3.11+** — everything server-side, including the existing `99-tools/`
- **[Chainlit](https://github.com/Chainlit/chainlit) 2.11+** — the entire
  frontend. Chainlit provides: an ASGI server (FastAPI under the hood — we
  don't write FastAPI code ourselves), password and OAuth auth, signed
  session cookies, streaming chat messages, tool-call step display
  (`cl.Step`), action buttons (`cl.AskActionMessage`), user-input prompts
  (`cl.AskUserMessage`), mermaid/plotly/pyplot/image/file/custom elements,
  a sidebar (`cl.ChatSettings`), and optional SQLite-backed chat history.
  Running `python -m chainlit run main.py` is what boots the service.
- **claude-agent-sdk** — Claude Code subprocess management. The server
  registers a **pre-tool-call hook** for the SDK's Bash tool — this is the
  preferred enforcement mechanism for checkpoint B (see **Action Interception**).
- **`claude` CLI** — invoked by the SDK in `cli` mode. Installed via
  `npm install -g @anthropic-ai/claude-code`. Imp's Setup Agent verifies it.
- **`anthropic` Python SDK** — used in `bare` mode (when the `claude` CLI cannot
  be installed in the deployment environment, e.g. a locked-down VPS without
  Node). `bare` mode talks to the Anthropic API directly with
  `ANTHROPIC_API_KEY`. Same model selection, same prompts, no Bash tool — which
  also means no `99-tools/*.py` invocation, so `bare` mode is **read-and-render
  only**: it can render charts but not write code or open PRs. Opt in via the
  Setup Agent.
- **gh CLI** — all GitHub operations. PyGithub is intentionally not used:
  one canonical path keeps the interception layer simple.
- **Mermaid** — chart rendering. Chainlit renders mermaid fenced code blocks
  in chat messages natively. Gantt, flowchart, kanban (as a flowchart), and
  burndown (as a quickchart image) all come out of mermaid syntax, so no
  Chart.js / Frappe Gantt / Jinja2 chart templates are needed. The `pipeline/`
  chart scripts generate mermaid source strings rather than HTML.
- **No React, no JS/HTML written by us.** Chainlit has a React frontend
  internally, but it's a pre-built static asset we never touch. We write
  Python handlers against Chainlit's Python API. There is no `ui/` directory,
  no bundler, no static files to serve ourselves, no WebSocket protocol to
  design.

dependency — already done, do not rewrite. The adoption of Chainlit shrank
this from 25 steps to 17: nine UI-renderer / server-plumbing issues were
obsoleted (see **Obsolete from the pre-Chainlit plan** at the end).

**Phase 1 — minimal chat loop with auth**

1. **imp.py** — Entry point: venv bootstrap, dep install, launch
   `python -m chainlit run main.py`. Generates `CHAINLIT_AUTH_SECRET` on
   first run.
2. **main.py** — The Chainlit app. Password auth callback,
   `on_chat_start` dispatch to setup vs. foreman handler, `on_message`
   stub that echoes. No security layer yet.

→ At this point you can `python imp.py`, log in, and echo-chat with a stub.
Verify before continuing.

**Phase 2 — security spine**

3. **server/intercept.py** — Action queue, command classifier (read vs.
   write), stub guard that auto-approves. This goes in *before* the worker
   has any tools, so the seam exists from day one.
4. **server/guard.py** — Real Guard Agent, both checkpoints A and B.
   Replaces the stub.
5. **server/budgets.py** — Token + edit + task counters backed by
   `.imp/state.json`. Hooked into `intercept.py`.

**Phase 3 — Setup Agent (configures everything else)**

6. **server/setup_agent.py** — Setup-mode agent + its tools
   (`set_admin_password`, `gh_auth_login`, `claude_auth_login`,
   `list_repos`, `set_repo`, `list_projects`, `create_imp_project`,
   `mark_setup_complete`, `configure_loop`). Wired into `main.py`'s
   `on_chat_start` so a missing/incomplete `.imp/config.json` routes the
   session to the setup handler. Uses `cl.AskUserMessage`, `cl.AskActionMessage`,
   and `cl.Step` for the conversational flow.
7. **pipeline/project_bootstrap.py** + **templates/fields.json** — Used by
   `create_imp_project`.

→ At this point a fresh clone can be configured end-to-end through the browser.

**Phase 4 — Foreman (the worker) and its visibility tools**

8. **server/foreman_agent.py** — Worker agent definition + tool registry,
   loaded after `setup_complete` flips. Uses the SDK pre-tool-call hook to
   funnel everything through `intercept.py`. Emits `cl.Step` for every
   tool call so tool usage is visible in the chat.
9. **pipeline/sync_issues.py** — GitHub issue + project field sync.
10. **pipeline/heuristics.py** — Inference (durations, dependencies, delays).
11. **pipeline/render_chart.py** + **templates/gantt.mmd.j2** — First chart,
    emitted as a mermaid source string wrapped in a fenced code block inside
    `cl.Message`.
12. **pipeline/scenario.py** — A/B comparison (mermaid).
13. **Remaining mermaid templates** — kanban, burndown, comparison.

**Phase 5 — wiring 99-tools as worker tools**

14. **`99-tools/_state.py` shim** — Make the legacy state module read/write
    `.imp/state.json` via `server/budgets.py` so chat-set budgets and CLI
    runs share counters.
15. **Foreman tool registry: code-writing pipeline** — Add
    `moderate_issues.py`, `solve_issues.py`, `fix_prs.py`, `run_all.sh` to
    `foreman_agent.py`'s tool list. Each invocation goes through
    `intercept.py` as a single proposed action surfaced via
    `cl.AskActionMessage` for the human approval gate.

**Phase 6 — autonomous loop**

16. **server/loop.py** — Scheduler, scope, pause, ticks. Triggers Foreman
    with a synthetic intent. Fully gated by the same Guard Agent. Emits
    updates via `cl.ChatSettings` and status messages.

**Phase 7 — polish**

17. **docs/README.md** + **.chainlit/config.toml** — Non-developer setup
    guide (password recovery, recommended Caddyfile for HTTPS) and a
    committed Chainlit config with the Imp theme, project name, and layout.

At each step, test that the previous step works before moving on. The split
between phase 1 (chat shell) and phase 2 (security spine before tools) is
deliberate: the interception seam must exist before any tool is wired in,
otherwise it gets retrofitted badly.

### Obsolete from the pre-Chainlit plan

The following were in the original build order and are no longer needed:

- `server/auth.py` — replaced by `@cl.password_auth_callback` in `main.py`
- `server/app.py` — Chainlit *is* the FastAPI app
- `ui/index.html`, `ui/app.js`, `ui/style.css` — no HTML/JS written by us
- `ui/renderers/markdown.js` — Chainlit renders markdown natively
- `ui/renderers/chart.js` — mermaid in fenced code blocks
- `ui/renderers/diff.js` — diff fenced code blocks
- `ui/renderers/selection.js` — `cl.AskActionMessage`
- `ui/renderers/proposed_action.js` — `cl.AskActionMessage` + `cl.Step`
- `ui/renderers/loop_status.js` — `cl.ChatSettings` sidebar
- Custom WebSocket message protocol — Chainlit owns the wire

## Testing

After each pipeline script, write a minimal test in `tests/` that runs it
against a fixture JSON file in `tests/fixtures/`. No pytest magic needed,
just scripts that assert and exit 0 or 1.

## What NOT To Do

- Do not write HTML, CSS, or JavaScript yourself. Chainlit owns the frontend.
  Everything the user sees is produced by Python handlers in `main.py` calling
  Chainlit primitives. If a UI need doesn't map cleanly onto a Chainlit
  primitive, either (a) work around it with `cl.CustomElement` (which does
  take a JSX file — use as a last resort), or (b) accept the layout Chainlit
  gives you. Do not add a second frontend.
- Do not use a database of your own. Files in `.imp/` and the GitHub Projects
  v2 board are the state. Chainlit has an optional SQLite persistence layer
  for chat history — it may be enabled but never treated as a source of truth.
  No Postgres / Redis.
- Do not build multi-user authentication, user management, or roles. Imp's web
  service is a **single-admin** app — exactly one password set during setup,
  hashed and stored in `.imp/config.json`, gating all access. No user table, no
  signup flow, no password reset email. If the admin forgets the password,
  they SSH into the host and delete the hash from config.
- Do not over-abstract. One file per concern, no class hierarchies.
- Do not use Docker for development. Docker is for distribution only.
- Do not hardcode repo names. Everything comes from `.imp/config.json`.
- Do not add CLI flags to `imp.py` for things that should be set in chat
  (budgets, loop config, repo). The terminal does dependency install and
  server start. Nothing else.
- Do not reimplement what `99-tools/` already does. Wrap it. The whole point of
  the merge is one implementation of the code-writing pipeline, called from
  two front doors (CLI and chat).
- Do not let the worker call `gh` or pipeline scripts directly without going
  through `server/intercept.py`. The whole security argument depends on a
  single chokepoint. If you find yourself adding a "trusted" path that bypasses
  the interceptor, you have introduced a vulnerability — fix the interceptor
  instead.
- Do not gate every gh subcommand inside a pipeline script individually. The
  trust unit is the script invocation. Approve once, run to completion.
- Do not bypass the guard for "small" or "obvious" actions. The whole value of
  the guard is that there are no exceptions. If a class of action is genuinely
  too cheap to gate, encode it as an auto-approve rule inside the guard, not
  as a bypass in the worker.
- Do not write structured PM data into issue bodies. It goes in Projects v2
  custom fields.
